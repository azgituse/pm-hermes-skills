# PRD文档创建规范

## 概述

本技能用于创建符合特定格式要求的PRD（产品需求文档），适用于低代码平台营销活动玩法组件的产品需求文档编写。

## 触发条件

当用户需要：
1. 创建新的玩法组件PRD文档
2. 参考已有PRD示例创建新需求文档
3. 生成PRD的Markdown版本用于飞书文档

## 核心要求

### 1. PRD文档结构

PRD必须包含以下章节：
- 变更记录
- 背景介绍（业务背景、业务价值）
- 产品概述（产品定位、服务对象）
- 需求列表
- 需求详情（核心章节）
- 非功能需求（性能、安全、系统）
- 关联历史需求
- 原型文件位置

### 2. 大表格格式要求（关键）

**第3章 - B端组件配置**：
- 列数：5列
- 列名：原型 | 模块 | 字段 | 说明 | 编辑活动页时是否可修改
- 第一列原型只在每个模块的第一个字段显示

**第4章 - 玩法配置弹窗**：
- 列数：6列  
- 列名：原型 | 步骤 | 字段分类 | 字段 | 说明 | 编辑活动时是否可修改
- 按步骤1/2/3组织
- 奖池配置必须详细展开，包括每行列的定义

**第6章 - 数据统计**：
- 列数：3列
- 列名：原型 | 模块 | 说明
- Tab页内容都写在说明列里

### 3. C端埋点需求格式

必须使用标准事件名（不可自创）：
- `PageShow` - 页面曝光
- `ComponentShow` - 组件曝光
- `ComponentPopupShow` - 组件弹窗曝光
- `ComponentClick` - 组件点击
- `ComponentPopupClick` - 组件弹窗点击

标准属性名：
- `newactivity_business_id` - 页面业务ID
- `component_type` - 组件类型（枚举新增：xxx组件）
- `component_id` - 组件ID
- `component_name` - 组件名称
- `component_part` - 组件点击位置（枚举具体按钮）
- `popup_type` - 组件弹窗类型（枚举弹窗类型）
- `popup_button` - 组件弹窗按钮（枚举按钮名称）

### 4. 玩法组件原型要求

必须创建4个原型文件：
1. **C端交互稿** - 展示组件各状态交互
2. **B端配置界面原型** - 展示编辑器右侧配置面板
3. **玩法配置弹窗原型** - 展示玩法创建/编辑弹窗
4. **数据统计页面原型** - 展示数据查看页面

### 5. 原型文件位置声明

必须在PRD末尾添加章节：
```markdown
## 原型文件位置

- C端交互稿：`[路径]`
- B端配置界面原型：`[路径]`
- 玩法配置弹窗原型：`[路径]`
- 数据统计页面原型：`[路径]`
```

## 工作流程

1. **学习参考示例**：仔细分析用户提供的参考PRD，特别是表格格式和埋点规范
   - **关键：逐行对照，不可凭记忆发挥**
   - **关键：注意表格列数，第3章5列、第4章6列、第6章3列**
   - **关键：理解这是迭代项目，不是从0到1，要严格复用历史格式**
2. **创建原型文件**：先完成4个HTML原型文件
   - 与运营确认每个字段和交互
   - 确保原型是最终版本后再写PRD
3. **编写PRD内容**：
   - 先写玩法简介（规则、奖池、发放逻辑、奖励类型、状态流转）
   - 再写C端页面流程和异常场景
   - 然后按大表格格式写第3、4、6章（**逐行对照参考文档**）
   - 最后写埋点需求（使用标准事件名）
4. **生成Markdown文件**：保存为.md文件
5. **创建飞书文档**：使用lark-cli创建并更新文档
6. **根据反馈迭代**：
   - 用户指出问题时，先理解问题本质
   - 不要辩解，直接修正
   - 确认理解是否正确

## 用户协作要点（重要）

### 理解"参考"的真正含义
- 用户说"参考猜拳PRD" = 要逐行对照复制格式，不是大概模仿
- 用户说"完全参考" = 格式必须一模一样，只替换内容
- 迭代项目中，用户的"历史上下文"比我的"优化建议"更重要

### 沟通中的自我修正
- 当用户指出问题时，先理解问题本质（是格式问题还是内容问题）
- 不要辩解，直接修正
- 修正后向用户确认理解是否正确
- 把用户的纠正记录下来，避免重复犯错

### 运营视角的思考
把自己代入运营角色：
- 运营一天配置多个活动，界面要高效
- 运营不懂技术，配置要直观
- 考虑误操作防护（如二次确认）
- 空间效率：表格布局比展开式更省垂直空间

## 关键原则：迭代项目 vs 从0到1项目

### 迭代项目（如报名玩法、猜拳玩法等）

**必须严格遵守：**
1. **严格参考历史PRD格式** - 当用户说"参考"或"完全参考"时，意味着要忠实还原历史文档的结构、表格列数、字段命名，而不是"优化"或"改进"
2. **PRD是对原型的文字还原** - 原型里有什么就写什么，不要脑补添加（如原型没有关闭按钮，埋点里就不应该有）
3. **格式统一 > 个人优化** - 团队长期磨合下来的标准格式，统一性比"我觉得更好"更重要
4. **逐行对照，不可凭记忆** - 必须打开参考文档逐行对照，不能凭印象发挥
5. **运营体验优先** - B端配置界面要考虑运营实际操作效率，空间效率很重要（如用表格布局、颜色选择器不显示hex码节省空间）

### 从0到1项目

可以适当发挥创新，但仍需遵循基本的产品文档规范。

## 常见错误（避免）

1. ❌ 将第3、4、6章写成小表格或分节形式
2. ❌ 自创事件英文名（如baoming_form_submit）
3. ❌ 漏掉原型文件位置声明
4. ❌ 玩法配置表格用5列而非6列
5. ❌ 漏掉主按钮配置的提示文案
6. ❌ popup_type/popup_button枚举值不完整
7. ❌ **在迭代项目中自由发挥格式** - 这是本次合作最重要的教训
8. ❌ **脑补添加原型中没有的元素** - 必须严格对照原型
9. ❌ **忽视运营操作体验** - B端配置要考虑运营实际使用场景，空间效率很重要
10. ❌ **颜色选择器显示hex码** - 为了节省空间，不显示hex码（运营能看到颜色即可）
11. ❌ **不理解"参考"的含义** - "参考"=逐行对照复制，不是大概模仿

## 验证清单

提交前检查：
- [ ] 第3章是5列表格
- [ ] 第4章是6列表格
- [ ] 第6章是3列表格
- [ ] 埋点使用标准事件名（PageShow/ComponentShow等）
- [ ] 包含原型文件位置章节
- [ ] 所有配置项都有"编辑活动时是否可修改"列

## 工具使用

```bash
# 创建飞书文档
lark-cli docs +create --title="[标题]" --markdown="@[文件].md"

# 更新飞书文档（覆盖模式）
lark-cli docs +update --doc="[doc_id]" --mode=overwrite --markdown="@[文件].md"
```
